# 用例遗漏检查规范

## 检查目标

把 AI 从“生成者”切换成“评审者”，对已有用例初稿做反向检查。

## 必查维度

| 维度 | 检查问题 |
| --- | --- |
| 主流程 | 是否覆盖用户最常用的成功路径 |
| 分支流程 | 是否覆盖不同业务条件下的不同结果 |
| 异常输入 | 是否覆盖空值、非法值、格式错误、超范围 |
| 边界值 | 是否覆盖等于、刚小于、刚大于边界 |
| 权限角色 | 是否覆盖无权限、越权、角色差异 |
| 状态流转 | 是否覆盖合法状态和非法状态 |
| 重复操作 | 是否覆盖重复领取、重复提交、重复使用 |
| 并发场景 | 是否覆盖并发点击、并发使用、额度竞争 |
| 数据一致性 | 前端、后端、订单、账户、日志是否一致 |
| 逆向流程 | 是否覆盖取消、回退、撤销、恢复 |
| 用例质量 | 是否重复、过细、过粗、不可执行 |

## 输出格式

```markdown
## 遗漏检查与补充场景

| 遗漏点 | 补充场景 | 风险说明 | 是否建议纳入 | 优先级 |
| --- | --- | --- | --- | --- |
```

## 采纳原则

- AI 提出的遗漏只作为建议。
- 测试人员必须判断场景是否真实存在。
- 当前版本不涉及的内容可以进入“待确认”或“后续版本关注”。
- 涉及资损、权限、数据一致性的遗漏优先纳入。
