# 智能订单履约与售后协同系统——第二部分需求评审与测试点矩阵

> 文档定位：第二部分课程的预置输出，用于承接第三部分“测试用例生成与优化”。  
> 评审原则：仅把需求原文明确内容写成确定结论；规则缺口统一标记为“待确认”。

## 一、需求摘要

| 项目 | 内容 |
| --- | --- |
| 业务目标 | 保证多渠道订单从创建、库存锁定、支付、发货到售后的状态、资金和库存一致，并降低人工售后判断成本。 |
| 功能范围 | 下单、库存锁定、支付回调、发货、售后申请、退款审批、系统定时任务、操作日志与通知。 |
| 适用终端 | Web 管理端、H5 用户端、开放 API。 |
| 核心角色 | 普通用户、商家运营、财务人员、平台管理员、系统任务。 |
| 主链路 | 创建订单 → 锁定库存 → 支付成功 → 发货 → 收货完成 → 申请售后 → 审核与退款。 |
| 核心风险 | 状态竞争、库存超卖或未释放、重复支付/退款、金额不一致、越权操作、异步任务重复执行。 |
| 需求优先级 | P0。 |

## 七、人工审核提示

- 优先人工确认 Q-06、Q-08、Q-09、Q-11、Q-14、Q-16；这些问题会直接改变 P0 测试点的预期结果。
- TP-024、TP-033、TP-052 不能只做页面演示，需要通过并发请求、故障注入或日志/数据库观测验证。
- AI 可以扩展边界和异常场景，但不能自行决定支付与取消冲突优先级、退款分摊公式或失败补偿策略。
- P0/P1 是基于需求中的资金、库存、状态和权限风险给出的课堂初稿，真实项目还应结合历史缺陷、调用量和线上损失重新排序。

## 九、课堂讲评建议

- 不逐条讲 60 个测试点，只展示“规则可追溯”“待确认标记”“风险优先级”三个特征。
- 用 TP-024 演示：一个测试点如果没有并发时序、可观察数据和最终一致性断言，就还不能直接执行。
- 用 TP-048/TP-050 演示：边界值不是机械写 `-1/0/1`，必须围绕业务门槛和金额精度设计。
- 用 Q-09 演示：AI 发现规则缺口后应提出问题，而不是替产品决定补偿策略。
