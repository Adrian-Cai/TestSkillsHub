# 审查实践

## 多模型审查模式

使用不同模型进行不同审查视角：

```
模型 A 编写代码
    │
    ▼
模型 B 审查正确性和架构
    │
    ▼
模型 A 处理反馈
    │
    ▼
人类做出最终决定
```

这捕获了单个模型可能遗漏的问题——不同模型有不同的盲点。

**审查 agent 的示例提示：**
```
审查此代码变更以确保正确性、安全性和对我们项目约定的遵循。
规范说 [X]。变更应该 [Y]。
将任何问题标记为关键、重要或建议。
```

## 死代码卫生

在任何重构或实现变更后，检查孤立代码：
1. 识别现在不可达或未使用的代码
2. 明确列出它
3. **删除前询问：** "我应该删除这些现在未使用的元素吗：[列表]？"

不要留下死代码——它会混淆未来的读者和 agent。但不要默默删除你不理解的东西。有疑问时，询问。

```
已识别死代码：
- src/utils/date.ts 中的 formatLegacyDate() — 被 formatDate() 替换
- src/components/ 中的 OldTaskCard 组件 — 被 TaskCard 替换
- src/config.ts 中的 LEGACY_API_URL 常量 — 没有剩余引用
→ 可以安全删除这些吗？
```

## 审查速度

慢审查会阻碍整个团队。上下文切换到审查的成本小于强加给其他人的等待成本。

- **在一个工作日内响应** — 这是最大值，不是目标
- **理想节奏：** 在审查请求到达后不久响应，除非深处于专注编码中。典型的变更应该在一天内完成多轮审查
- **优先快速个人响应** 而不是快速最终批准。快速反馈即使需要多轮也能减少挫败感
- **大变更：** 要求作者拆分它们，而不是审查一个巨大的变更集

## 处理分歧

解决审查争议时，应用此层次结构：

1. **技术事实和数据** 覆盖意见和偏好
2. **样式指南** 是风格问题的绝对权威
3. **软件设计** 必须基于工程原则而非个人偏好进行评估
4. **代码库一致性** 如果不降低整体健康状况则是可接受的

**不要接受"我稍后会清理它。"** 经验表明延迟清理很少发生。要求在提交前清理，除非是真正的紧急情况。如果周围问题不能在此变更中解决，要求提交一个带有自我分配的 bug。

## 审查中的诚实

审查代码时——无论是你自己、另一个 agent 还是人类编写的：
- **不要橡皮图章。** 没有审查证据的 "LGTM" 对任何人都没有帮助。
- **不要淡化真正的问题。** 当它是一个将击中生产的 bug 时说"这可能是个小问题"是不诚实的。
- **尽可能量化问题。** "这个 N+1 查询将为列表中的每个项目增加约 50ms" 比"这可能很慢"更好。
- **对有明显问题的方法进行反驳。** 谄媚是审查中的失败模式。如果实现有问题，直接说出来并提出替代方案。
- **优雅地接受覆盖。** 如果作者有完整上下文并不同意，遵从他们的判断。评论代码，而不是人——将个人批评重新构建为专注于代码本身。

## 依赖纪律

代码审查的一部分是依赖审查：

**添加任何依赖之前：**
1. 现有技术栈解决这个问题吗？（通常可以。）
2. 依赖有多大？（检查包影响。）
3. 它是否积极维护？（检查最后提交、开放问题。）
4. 它是否有已知漏洞？（`npm audit`）
5. 许可证是什么？（必须与项目兼容。）

**规则：** 优先使用标准库和现有实用程序而不是新依赖。每个依赖都是负担。
