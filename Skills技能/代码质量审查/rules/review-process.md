# 审查过程

## 步骤 1：理解上下文

在查看代码之前，理解意图：
- 这个变更试图完成什么？
- 它实现了什么规范或任务？
- 预期的行为变更是什么？

## 步骤 2：先审查测试

测试揭示意图和覆盖范围：
- 变更是否有测试？
- 它们是否测试行为（而不是实现细节）？
- 边缘情况是否覆盖？
- 测试是否有描述性名称？
- 如果代码更改，测试是否会捕获回归？

## 步骤 3：审查实现

带着五个维度审视代码：

```
对于每个更改的文件：
1. 正确性：此代码是否做了测试所说它应该做的？
2. 可读性：我能在没有帮助的情况下理解这个吗？
3. 架构：这是否符合系统？
4. 安全性：有任何漏洞吗？
5. 性能：有任何瓶颈吗？
```

## 步骤 4：分类发现

为每个评论标记其严重性，以便作者知道什么是必需的 vs 可选的：

| 前缀 | 含义 | 作者操作 |
|------|------|----------|
| *（无前缀）* | 必需的更改 | 合并前必须解决 |
| **关键：** | 阻止合并 | 安全漏洞、数据丢失、功能损坏 |
| **Nit：** | 小的，可选的 | 作者可以忽略——格式化、风格偏好 |
| **可选：** / **考虑：** | 建议 | 值得考虑但不是必需的 |
| **FYI** | 仅供参考 | 无需操作——供将来参考的上下文 |

这防止作者将所有反馈视为强制性的，并在可选建议上浪费时间。

## 步骤 5：验证验证

检查作者的验证故事：
- 运行了哪些测试？
- 构建是否通过？
- 变更是否手动测试？
- UI 变更是否有截图？
- 是否有前后比较？
