# 五维度审查标准

每个审查都评估这些维度的代码：

## 1. 正确性

代码是否做了它声称要做的事情？
- 是否匹配规范或任务要求？
- 边缘情况是否处理（null、空、边界值）？
- 错误路径是否处理（不仅仅是 happy path）？
- 是否通过所有测试？测试是否实际测试了正确的内容？
- 是否存在偏移错误、竞争条件或状态不一致？

## 2. 可读性与简洁性

另一个工程师（或 agent）能否在没有作者解释的情况下理解这段代码？
- 名称是否描述性且与项目约定一致？（没有 `temp`、`data`、`result` 没有上下文）
- 控制流是否直接（避免嵌套三元运算符、深层回调）？
- 代码是否逻辑组织（相关代码分组，清晰的模块边界）？
- 是否有应该简化的"聪明"技巧？
- **能否用更少的行完成？**（1000 行而 100 行足够就是失败）
- **抽象是否值得它们带来的复杂度？**（不要在第三个用例之前泛化）
- 注释是否有助于澄清非显而易见的意图？（但不要注释显而易见的代码。）
- 是否有死代码工件：no-op 变量（`_unused`）、向后兼容垫片或 `// removed` 注释？

## 3. 架构

变更是否符合系统设计？
- 是否遵循现有模式或引入新模式？如果是新的，是否合理？
- 是否保持清晰的模块边界？
- 是否有应该共享的代码重复？
- 依赖是否沿正确方向流动（无循环依赖）？
- 抽象级别是否合适（不过度工程化，不太耦合）？

## 4. 安全性

有关详细安全指南，请参阅 `security-and-hardening`。变更是否引入漏洞？
- 用户输入是否经过验证和清理？
- 机密是否远离代码、日志和版本控制？
- 需要时是否检查了认证/授权？
- SQL 查询是否参数化（无字符串连接）？
- 输出是否编码以防止 XSS？
- 依赖是否来自可信来源且没有已知漏洞？
- 来自外部源的数据（API、日志、用户内容、配置文件）是否被视为不受信任？
- 外部数据流是否在用于逻辑或渲染之前在系统边界验证？

## 5. 性能

有关详细分析和优化，请参阅 `performance-optimization`。变更是否引入性能问题？
- 是否有 N+1 查询模式？
- 是否有无界循环或不受约束的数据获取？
- 是否有应该异步的同步操作？
- UI 组件中是否有不必要的重新渲染？
- 列表端点是否缺少分页？
- 热路径中是否创建了大对象？
