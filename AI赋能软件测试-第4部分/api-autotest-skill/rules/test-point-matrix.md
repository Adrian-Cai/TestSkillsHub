# 测试点矩阵生成规范

## 目标

将测试要素转化为结构化、可追溯、按优先级分层的测试点矩阵。

## 标准输出格式

| 测试类别 | 测试点 | 输入数据 | 预期结果 | 优先级 | 是否适合自动化 |
| -------- | ------ | -------- | -------- | ------ | -------------- |
| 正常流程 |        |          |          | P0     | 是             |
| 参数异常 |        |          |          | P1     | 是             |
| 业务异常 |        |          |          | P0/P1  | 是             |
| 鉴权异常 |        |          |          | P0     | 是             |
| 安全风险 |        |          |          | P1     | 部分适合       |
| 待确认   |        |          | 待确认   | -      | 否             |

## 优先级分层规则

### P0 - 必须执行

以下场景必须标注 P0：

- 接口主流程（正常成功路径）。
- 核心业务规则校验（如重复注册、余额不足、权限拦截）。
- 鉴权完全缺失（无 token）。
- 会造成数据异常或安全漏洞的场景。

### P1 - 应该执行

以下场景标注 P1：

- 常见的必填参数缺失。
- 常见格式错误（邮箱格式、手机号格式）。
- 参数边界值（最大、最小、超限）。
- token 过期、权限不足等鉴权异常。
- 接口错误提示是否符合预期。

### P2 - 可以执行

以下场景标注 P2：

- 低频兼容性场景。
- 非核心字段的类型错误。
- 非关键组合场景。
- 已被 P0/P1 覆盖的变体场景。

### 待确认 - 不进入脚本

以下情况进入「待确认」区域：

- 业务规则文档未明确说明。
- 返回结构文档未完整定义。
- 依赖其他接口或系统的场景。
- 需要和业务方确认的边界情况。

## 自动化适合性判断

| 适合自动化 | 判断依据 |
| ---------- | -------- |
| 是 | 输入确定、预期结果可断言、可重复执行、不依赖人工判断 |
| 部分适合 | 主要流程可自动化，但某些断言需人工验证 |
| 否 | 预期结果不确定、依赖外部系统状态、需要人工观察 |

## 测试点命名规范

测试点名称必须：

- 包含被测对象（哪个参数/哪个业务规则）。
- 包含测试动作（缺失/为空/格式错误/超长/重复等）。
- 包含预期行为（返回什么错误/成功）。

示例（好）：
- `用户名为空时，返回参数缺失错误`
- `密码长度超过32位时，返回参数格式错误`
- `无 token 访问需鉴权接口，返回 401`
- `正常用户名密码登录，返回 token 且不含密码字段`

示例（差）：
- `测试登录`
- `case1`
- `验证参数`

## 测试点与要素的追溯关系

每条测试点必须能追溯到：

- 接口资料表中的某个字段或业务规则。
- 测试要素拆解中的某个维度条目。

如果测试点无法追溯，说明它是脑补的，必须删除或移入「待确认」。

## 典型测试类别与数量参考

| 接口类型 | 建议最少测试点数 | 必覆盖类别 |
| -------- | ---------------- | ---------- |
| 登录接口 | 10+ | 正常流程、参数异常、鉴权、安全 |
| 注册接口 | 12+ | 正常流程、参数异常、唯一性、业务规则 |
| 查询接口 | 8+ | 正常流程、参数异常、鉴权、数据状态 |
| 创建/修改接口 | 10+ | 正常流程、参数异常、业务规则、鉴权 |
| 删除接口 | 8+ | 正常流程、业务规则、鉴权、数据状态 |
| 支付/下单接口 | 15+ | 正常流程、金额边界、状态限制、幂等、鉴权 |
| 文件上传接口 | 10+ | 正常流程、文件类型、大小限制、鉴权 |
