# 人工审查清单输出规范

## 目标

在脚本生成完成后，输出标准化人工审查清单，提醒测试工程师在实际执行前必须核对的关键项目。

## 标准审查清单格式

| 审查项                       | 是否通过  | 问题说明 |
| ---------------------------- | --------- | -------- |
| 接口路径是否正确              | 待人工确认 |          |
| 请求方法是否正确              | 待人工确认 |          |
| 请求参数是否完整              | 待人工确认 |          |
| 鉴权方式是否正确              | 待人工确认 |          |
| 测试数据是否可重复执行        | 待人工确认 |          |
| 是否存在硬编码账号密码        | 待人工确认 |          |
| 是否设置了 timeout            | 待人工确认 |          |
| 是否只断言 HTTP 200           | 待人工确认 |          |
| 是否校验业务 code/message     | 待人工确认 |          |
| 失败场景是否不返回 token      | 待人工确认 |          |

| 环境变量是否已配置说明        | 待人工确认 |          |

## 各审查项说明

### 接口路径是否正确

- AI 生成路径来自用户提供的文档，可能存在版本差异。
- 需确认路径与实际测试环境一致（含版本前缀如 `/api/v2/`）。

### 请求方法是否正确

- 确认 GET/POST/PUT/PATCH/DELETE 与文档一致。
- 注意部分系统用 POST 代替 DELETE。

### 请求参数是否完整

- 对比文档参数列表与脚本中的参数，确认无遗漏。
- 注意可选参数是否需要覆盖测试。

### 鉴权方式是否正确

- 确认 token 的传递方式：Header（Bearer/Token）、Cookie、参数。
- 确认 token 的获取方式：登录接口/环境变量/测试工具。

### 测试数据是否可重复执行

- 确认测试数据不会因为一次执行而消耗（如注册后账号已存在）。
- 注意创建类接口需要每次使用唯一标识（时间戳、UUID）。
- 注意删除类接口需要在每次执行前预先创建数据。

### 是否存在硬编码账号密码

- 检查代码中是否有明文账号、密码、token。
- 所有敏感信息必须通过环境变量注入。

### 是否设置了 timeout

- 每个 `requests` 调用必须有 `timeout` 参数。
- 推荐值：普通接口 10s，文件上传 60s，复杂计算 30s。

### 是否只断言 HTTP 200

- 高风险：只写 `assert response.status_code == 200`。
- 必须额外断言业务 code、关键字段、敏感字段缺失。

### 是否校验业务 code/message

- 确认每个测试用例都有对业务层的断言。
- 「待确认」项需在实际运行前补充。

### 失败场景是否不返回 token

- 所有失败场景（参数错误、鉴权失败、业务异常）都不应返回 token 或敏感数据。
- 脚本中需要显式断言此项。

### 是否存在敏感信息泄露风险

- 确认错误响应中不含密码、堆栈信息、内部路径、数据库表名。
- 确认日志输出不打印 token 和密码。

### 环境变量是否已配置说明

- `.env.example` 文件是否列出了所有必要的环境变量。
- README 是否说明了如何配置运行环境。

### 数据库/状态依赖是否已说明

- 确认测试运行所需的前置数据状态（如需要某用户已存在）。
- 确认测试结束后是否需要清理数据。

### fixture 作用域是否合理

- `scope="session"`：整个测试会话共享，适合 base_url、session。
- `scope="function"`：每个用例独立，适合需要隔离的数据。
- 避免 session 级 fixture 携带上一个用例的副作用。

## 待确认问题清单格式

在审查清单之后，必须附上待确认问题清单：

| 编号 | 问题描述 | 影响范围 | 负责确认人 | 截止时间 |
| ---- | -------- | -------- | ---------- | -------- |
| Q001 |          |          | 开发/业务  | -        |
| Q002 |          |          | 开发/业务  | -        |

## 审查清单输出时机

以下情况必须输出审查清单：

1. 每次生成 pytest 脚本后。
2. 对已有代码进行审查优化后。
3. 用例数据设计完成后（简化版）。

## AI 生成初稿的边界说明

必须在审查清单前声明：

```text
以上脚本为 AI 辅助生成的初稿，基于用户提供的接口信息起草。
测试工程师在实际执行前必须完成以下工作：
1. 逐项核对审查清单。
2. 确认所有「待确认」项目的实际业务规则。
3. 在真实测试环境中配置环境变量。
4. 执行一次冒烟测试验证脚本可运行。
5. 根据实际返回结果补充/修正断言逻辑。
```
