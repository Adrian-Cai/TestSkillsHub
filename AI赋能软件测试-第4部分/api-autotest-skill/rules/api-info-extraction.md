# 接口资料整理规范

## 目标

将用户提供的接口信息整理成标准化接口资料表，作为后续所有测试设计的唯一依据。

## 输出格式

| 项目     | 内容 | 备注      |
| -------- | ---- | --------- |
| 接口名称 |      |           |
| 请求方法 |      |           |
| 请求路径 |      |           |
| 请求头   |      |           |
| 请求参数 |      |           |
| 返回字段 |      |           |
| 错误码   |      |           |
| 鉴权方式 |      |           |
| 业务规则 |      |           |
| 待确认项 |      | 不允许脑补 |

## 处理原则

### 能从资料确定的

- 直接写出，不添加推测内容。
- 从 curl 命令中提取：URL、Method、Headers、Body。
- 从 Swagger 中提取：paths、parameters、requestBody、responses。
- 从 Apifox/Postman 中提取：接口名称、请求配置、示例数据。

### 不能确定的

- 统一写「待确认」，列入待确认问题清单。
- 不把示例值当作业务规则（示例是示例，不是约束）。
- 不把 HTTP 200 误判为业务成功（需看业务 code 字段）。
- 不把 AI 推测内容当成事实。

### 常见待确认项

- 业务 code 的完整枚举值。
- 参数长度、格式的具体约束（文档未明确时）。
- 鉴权 token 的获取方式和有效期。
- 重复提交的幂等策略。
- 并发场景的处理机制。
- 返回字段的完整结构（文档只给示例时）。

## curl 解析规则

输入 curl 命令时，提取以下信息：

```text
curl -X POST "https://api.example.com/login" \
  -H "Content-Type: application/json" \
  -H "X-Request-ID: abc123" \
  -d '{"username":"test","password":"123456"}'
```

提取结果：
- 请求方法：POST
- 请求路径：/login（域名部分记录为环境变量，不硬编码）
- 请求头：Content-Type: application/json，X-Request-ID（值脱敏）
- 请求体参数：username（string），password（string，敏感字段）

## 敏感信息处理

以下信息必须脱敏，不得原样保留：

- 真实域名 → 替换为 `${BASE_URL}`
- 真实账号 → 替换为 `${TEST_USER}`
- 真实密码 → 替换为 `${TEST_PASSWORD}`
- 真实 token → 替换为 `${AUTH_TOKEN}`
- 真实手机号/邮箱 → 替换为脱敏占位符
