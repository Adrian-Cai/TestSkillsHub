# 测试点拆解 Prompt 模板

## 使用场景

当测试人员已经拿到需求文档、需求摘要或业务规则清单，需要借助 AI 生成测试点矩阵初稿时使用。

## Prompt 模板一：从需求文档生成测试分析

```text
你是一名资深软件测试分析师。

请基于以下需求文档，帮助我完成需求评审和测试点拆解。

【需求文档】
<粘贴需求内容>

请按以下结构输出：
1. 需求摘要
   - 业务目标
   - 涉及用户角色
   - 功能范围
   - 主业务流程
   - 不在本次范围内的内容

2. 业务规则清单
   - 业务规则
   - 字段规则
   - 权限规则
   - 状态规则
   - 时间规则
   - 金额 / 数量 / 次数限制
   - 异常处理规则

3. 测试点矩阵
   字段包括：
   - 模块
   - 规则 / 场景
   - 测试点
   - 测试类型
   - 优先级
   - 风险说明

4. 需求疑问与待确认项

要求：
- 不要补充需求中没有明确说明的规则。
- 如果存在不确定内容，请标记为“待确认”。
- 测试点需要覆盖正常流、异常流、边界值、权限、状态变化、重复提交和数据一致性。
- AI 输出只作为初稿，需方便测试人员人工审查。
```

## Prompt 模板二：从规则清单生成测试点矩阵

```text
你是一名资深功能测试工程师。

下面是已经整理好的需求摘要和业务规则清单，请基于这些内容生成测试点矩阵。

【需求摘要】
<粘贴需求摘要>

【业务规则清单】
<粘贴规则清单>

请输出测试点矩阵，字段包括：
| 模块 | 规则/场景 | 测试点 | 测试类型 | 优先级 | 风险说明 |

测试类型至少覆盖：
- 正常流
- 异常流
- 边界值
- 权限校验
- 状态流转
- 重复提交 / 幂等
- 数据一致性
- 提示文案

优先级请使用：高 / 中 / 低。

请注意：
- 只生成与当前需求强相关的测试点。
- 不确定的业务规则请放入“待确认项”，不要当作已确定规则。
- 每个测试点应能继续转化为功能测试用例。
```

## Prompt 模板三：让 AI 做补漏检查

```text
请对以下测试点矩阵做补漏检查。

【需求摘要】
<粘贴需求摘要>

【业务规则清单】
<粘贴规则清单>

【测试点矩阵】
<粘贴测试点矩阵>

请检查：
1. 是否遗漏异常流程
2. 是否遗漏边界值
3. 是否遗漏权限或角色差异
4. 是否遗漏状态流转
5. 是否存在需求歧义
6. 是否存在规则冲突
7. 是否存在高风险但未标记的场景

输出格式：
| 问题类型 | 问题描述 | 影响范围 | 建议处理方式 | 是否需要产品确认 |

要求：
- 明确区分“确定问题”和“待确认问题”。
- 不要直接修改测试点矩阵，先给出审查意见。
```

## 人工审查要求

使用以上 Prompt 后，测试人员必须检查：

- AI 是否把未说明的内容当成确定规则
- AI 生成的测试点是否符合真实业务
- 是否存在无效、过度泛化或与当前需求无关的测试点
- 是否需要补充历史缺陷、线上风险或团队经验
- 高风险测试点是否需要进入评审会议或测试计划
