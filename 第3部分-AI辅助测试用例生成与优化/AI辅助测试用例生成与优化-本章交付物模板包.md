# AI 辅助测试用例生成与优化：本章交付物模板包

## 1. 本章可交付物清单

| 交付物 | 使用时机 | 主要作用 | 责任边界 |
|---|---|---|---|
| AI 辅助测试用例生成流程模板 | 开始测试设计前 | 建立“需求输入 - AI处理 - 人工审查 - 测试产出”的工作流程 | AI 辅助整理与扩展，测试人员负责确认 |
| 业务规则拆解提示词模板 | 拿到需求后 | 先拆规则，避免直接生成用例导致覆盖失真 | 测试人员审查业务理解是否正确 |
| 按测试设计方法生成用例提示词模板 | 规则拆解后 | 用测试方法约束 AI 输出，形成结构化用例初稿 | 测试人员判断方法适配性和场景真实性 |
| 用例遗漏检查清单 | 用例初稿完成后 | 反向检查异常、边界、权限、状态、数据一致性等遗漏 | 测试人员决定是否纳入测试范围 |
| 用例裁剪与合并清单 | 用例数量较多时 | 删除低价值用例，合并重复用例，统一粒度 | 测试人员负责最终用例集质量 |
| 用例优先级判断清单 | 制定执行计划前 | 按风险、频率、影响面确定 P0/P1/P2 | 测试人员结合版本目标和资源判断 |
| 优惠券案例优化前后对比样例 | 课堂演示 / 课后练习 | 让学员看到三轮迭代后的质量差异 | 用作示例，不作为唯一标准答案 |

## 2. AI 辅助测试用例生成流程模板

### 流程总览

| 步骤 | 输入 | AI 处理 | 人工审查 | 最终产出 |
|---|---|---|---|---|
| 1. 需求输入 | 需求说明、原型、用户故事、接口说明、业务规则 | 读取并总结需求信息 | 判断需求范围是否完整 | 需求输入材料包 |
| 2. 业务规则拆解 | 原始需求材料 | 提取角色、对象、规则、限制、状态、异常、疑问点 | 检查 AI 是否误解业务，补充遗漏规则 | 业务规则拆解清单 |
| 3. 方法化生成用例 | 已确认的业务规则 | 按场景法、等价类、边界值、判定表、状态迁移法生成用例 | 判断场景是否成立，删除不符合业务实际的内容 | 测试用例初稿 |
| 4. 反向检查遗漏 | 原始需求 + 用例初稿 | 从异常、边界、权限、状态、并发、数据一致性等角度检查 | 判断遗漏是否真实、是否值得覆盖 | 遗漏场景补充清单 |
| 5. 裁剪与合并 | 用例初稿 + 补充场景 | 标识重复、相似、粒度异常的用例 | 删除、合并、调整粒度 | 可执行测试用例集 |
| 6. 优先级排序 | 可执行测试用例集、业务风险、历史缺陷 | 给出 P0/P1/P2 建议 | 结合版本目标、资源和风险最终确认 | 带优先级的最终用例集 |

### 推荐用例字段

| 字段 | 说明 |
|---|---|
| 用例编号 | 便于追踪和评审 |
| 用例标题 | 用一句话表达测试目标 |
| 测试设计方法 | 场景法 / 等价类 / 边界值 / 判定表 / 状态迁移法 |
| 前置条件 | 执行前必须满足的条件 |
| 测试数据 | 关键输入数据 |
| 操作步骤 | 控制在可执行粒度 |
| 预期结果 | 可验证、可判断 |
| 优先级 | P0 / P1 / P2 |
| 备注 | 记录风险、依赖或待确认点 |


### 裁剪与合并判断表

| 判断项 | 处理建议 |
|---|---|
| 用例验证目标完全相同 | 合并 |
| 只有测试数据不同，但验证逻辑相同 | 合并为参数化用例 |
| 业务场景不成立 | 删除 |
| 与当前版本无关 | 移至后续版本或回归池 |
| 只验证低价值展示内容 | 降低优先级或合并 |
| 一个用例覆盖多个独立风险 | 拆分 |
| 步骤过长且失败定位困难 | 拆分 |
| 前置条件复杂但风险低 | 降低优先级 |
| 涉及核心链路或资损风险 | 保留 |
| 涉及历史高发缺陷 | 保留并提高优先级 |

## 8. 优惠券案例优化前后对比样例

### 案例需求简述

用户可以领取平台优惠券，并在下单时使用。每个用户每种优惠券限领一次。优惠券有使用门槛、有效期、适用商品范围。订单取消后优惠券是否退回由优惠券规则决定。过期、已使用、未达到门槛、商品不适用时不可使用。

---

### 第一轮：直接生成后的典型结果

| 用例编号 | 用例标题 | 主要问题 |
|---|---|---|
| TC-001 | 用户成功领取优惠券 | 只覆盖主流程 |
| TC-002 | 用户下单使用优惠券 | 未说明门槛、商品范围、有效期 |
| TC-003 | 用户重复领取优惠券 | 有异常，但缺少预期状态校验 |
| TC-004 | 优惠券过期不可使用 | 未覆盖边界时间 |
| TC-005 | 优惠券使用后状态变更 | 未覆盖订单取消后的状态 |

### 第一轮问题总结

| 问题类型 | 表现 |
|---|---|
| 规则拆解不足 | 没有系统拆出领取限制、使用门槛、适用范围、退券规则 |
| 边界不足 | 没有覆盖刚好达到门槛、刚好过期、有效期最后一秒 |
| 条件组合不足 | 没有覆盖金额、商品、有效期、状态的组合 |
| 状态不足 | 没有覆盖未领取、已领取、已使用、已过期、已退回 |
| 风险不足 | 没有突出重复使用、金额错误、资损风险 |

| TC-Coupon-002 | 场景法 | 用户领取后下单成功使用优惠券 | 订单金额满足门槛，商品适用 | 优惠金额正确抵扣，券状态为已使用 | P0 |
| TC-Coupon-003 | 等价类 | 已领取用户再次领取同一优惠券 | 同一用户重复领取 | 领取失败，提示已领取 | P1 |
| TC-Coupon-004 | 等价类 | 商品不在适用范围时不可使用 | 商品不适用 | 优惠券不可选或使用失败 | P1 |

## 9. 本章使用原则

- AI 生成得快，不代表测试覆盖准确。
- 先拆规则，再生成用例。
- 用测试设计方法约束 AI 输出。
- 让 AI 反向检查遗漏，但由测试人员决定是否采纳。
- 最终测试用例集必须经过人工裁剪、合并和优先级排序。
- AI 辅助测试设计，测试人员承担质量责任。
