# 业务规则拆解规范

## 拆解目标

先确认“测什么”，再生成用例。不要一开始直接生成测试用例。

## 必须拆解的内容

| 维度 | 检查问题 |
| --- | --- |
| 业务对象 | 功能围绕哪些对象展开，例如订单、优惠券、用户、账户 |
| 用户角色 | 谁可以操作，是否存在角色或权限差异 |
| 前置条件 | 执行功能前必须满足什么状态或数据条件 |
| 主流程 | 用户最常走的成功路径是什么 |
| 分支流程 | 不同条件下系统有什么不同表现 |
| 输入限制 | 金额、数量、时间、长度、格式、枚举值 |
| 状态变化 | 状态从哪里来，到哪里去，哪些状态禁止 |
| 异常规则 | 过期、重复、为空、失效、不满足条件时如何处理 |
| 权限约束 | 无权限、越权、未登录、Token 失效 |
| 数据依赖 | 是否依赖库存、额度、账户、配置、第三方结果 |
| 待确认问题 | 需求没有说清楚但影响测试设计的问题 |

## 输出格式

```markdown
## 业务规则拆解

### 明确规则
- ...

### 隐含规则
- ...

### 状态与流程
- ...

### 异常与限制
- ...

### 待确认问题
- [ ] ...
```

## 判断原则

- 需求明确写出的内容，放入“明确规则”。
- 可以合理推导但不确定的内容，放入“隐含规则”或“待确认问题”。
- 不要自行创造业务规则。
- 若规则影响资损、权限、数据一致性，必须标记为高关注点。
