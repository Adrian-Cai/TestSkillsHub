# 规划与任务拆解

## 概述

将工作分解为具有明确验收标准的小型、可验证任务。好的任务分解是可靠完成工作的 agent 和产生混乱的 agent 之间的区别。每个任务都应该足够小，可以在单个专注会话中实现、测试和验证。

## 何时使用

- 你有规范并需要将其分解为可实现单元
- 任务感觉太大或太模糊无法开始
- 工作需要跨多个 agent 或会话并行化
- 你需要向人类传达范围
- 实现顺序不明显

**何时不使用：** 范围明显的单文件更改，或规范已包含定义明确的任务。

## 规划过程

### 步骤 1：进入计划模式

在编写任何代码之前，以只读模式运行：
- 阅读规范和相关代码库部分
- 识别现有模式和约定
- 映射组件间的依赖
- 注意风险和未知数

**规划期间不要编写代码。** 输出是计划文档，而不是实现。

### 步骤 2：识别依赖图

映射什么依赖什么：

```
数据库模式
    │
    ├── API 模型/类型
    │       │
    │       ├── API 端点
    │       │       │
    │       │       └── 前端 API 客户端
    │       │               │
    │       │               └── UI 组件
    │       │
    │       └── 验证逻辑
    │
    └── 种子数据 / 迁移
```

实现顺序遵循依赖图自底向上：先构建基础。

### 步骤 3：垂直切片

而不是构建所有数据库，然后所有 API，然后所有 UI——一次构建一个完整的功能路径：

**坏（水平切片）：**
```
任务 1：构建整个数据库模式
任务 2：构建所有 API 端点
任务 3：构建所有 UI 组件
任务 4：连接一切
```

**好（垂直切片）：**
```
任务 1：用户可以创建帐户（注册的模式 + API + UI）
任务 2：用户可以登录（登录的认证模式 + API + UI）
任务 3：用户可以创建任务（创建的任务模式 + API + UI）
任务 4：用户可以查看任务列表（列表视图的查询 + API + UI）
```

每个垂直切片都提供可工作的、可测试的功能。

### 步骤 4：编写任务

每个任务遵循此结构：

```markdown
## 任务 [N]：[简短描述性标题]

**描述：** 一段话解释此任务完成什么。

**验收标准：**
- [ ] [具体、可测试的条件]
- [ ] [具体、可测试的条件]

**验证：**
- [ ] 测试通过：`npm test -- --grep "功能名称"`
- [ ] 构建成功：`npm run build`
- [ ] 手动检查：[要验证的内容]

**依赖：** [此任务依赖的任务编号，或"无"]

**可能触及的文件：**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**估算范围：** [小：1-2 个文件 | 中：3-5 个文件 | 大：5+ 个文件]
```

### 步骤 5：排序和检查点

排列任务以便：
1. 满足依赖（先构建基础）
2. 每个任务使系统处于可工作状态
3. 每 2-3 个任务后有验证检查点
4. 高风险任务在前面（快速失败）

添加明确检查点：

```markdown
## 任务 1-3 后的检查点
- [ ] 所有测试通过
- [ ] 应用程序无错误构建
- [ ] 核心用户流程端到端工作
- [ ] 继续前与人类审查
```

## 任务大小指南

| 大小 | 文件 | 范围 | 示例 |
|------|------|------|------|
| **XS** | 1 | 单函数或配置更改 | 添加验证规则 |
| **S** | 1-2 | 一个组件或端点 | 添加新 API 端点 |
| **M** | 3-5 | 一个功能切片 | 用户注册流程 |
| **L** | 5-8 | 多组件功能 | 带过滤和分页的搜索 |
| **XL** | 8+ | **太大——进一步分解** | — |

如果任务是 L 或更大，应分解为更小的任务。Agent 在 S 和 M 任务上表现最佳。

**何时进一步分解任务：**
- 需要超过一个专注会话（大约 2+ 小时的 agent 工作）
- 你无法用 3 个或更少的要点描述验收标准
- 它涉及两个或更多独立子系统（例如，认证和计费）
- 你在任务标题中发现自己在写"和"（它是两个任务的标志）

## 计划文档模板

```markdown
# 实现计划：[功能/项目名称]

## 概述
[我们构建什么的单段摘要]

## 架构决策
- [关键决策 1 及原因]
- [关键决策 2 及原因]

## 任务列表

### 阶段 1：基础
- [ ] 任务 1：...
- [ ] 任务 2：...

### 检查点：基础
- [ ] 测试通过，构建干净

### 阶段 2：核心功能
- [ ] 任务 3：...
- [ ] 任务 4：...

### 检查点：核心功能
- [ ] 端到端流程工作

### 阶段 3：完善
- [ ] 任务 5：...
- [ ] 任务 6：...

### 检查点：完成
- [ ] 所有验收标准满足
- [ ] 准备审查

## 风险和缓解
| 风险 | 影响 | 缓解 |
|------|------|------|
| [风险] | [高/中/低] | [策略] |

## 开放问题
- [需要人类输入的问题]
```

## 并行化机会

当有多个 agent 或会话可用时：
- **安全并行化：** 独立功能切片、已实现功能的测试、文档
- **必须顺序执行：** 数据库迁移、共享状态更改、依赖链
- **需要协调：** 共享 API 契约的功能（先定义契约，然后并行化）

## 常见误解

| 误解 | 现实 |
|------|------|
| "我会边走边弄清楚" | 这就是你最终陷入混乱和返工的原因。10 分钟的规划节省数小时。 |
| "任务是显而易见的" | 无论如何写下它们。明确的任务揭示隐藏的依赖和遗忘的边缘情况。 |
| "规划是开销" | 规划就是任务。没有计划的实现只是打字。 |
| "我可以在脑子里装下所有内容" | 上下文窗口是有限的。书面计划在会话边界和压缩后仍然存在。 |

## 红旗

- 在没有书面任务列表的情况下开始实现
- 说"实现功能"而没有验收标准的任务
- 计划中没有验证步骤
- 所有任务都是 XL 大小
- 任务间没有检查点
- 没有考虑依赖顺序

## 验证

开始实现之前，确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖已识别并正确排序
- [ ] 没有任务触及超过约 5 个文件
- [ ] 主要阶段之间存在检查点
- [ ] 人类已审查并批准计划
