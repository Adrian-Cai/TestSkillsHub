# 代码审查与质量

## 概述

使用质量门禁进行多维度代码审查。每个变更在合并前都要审查——无一例外。审查涵盖五个维度：正确性、可读性、架构、安全性和性能。

**批准标准：** 当变更明确改善整体代码健康状况时批准它，即使它不完美。完美的代码不存在——目标是持续改进。不要因为变更不是你编写的方式而阻止它。如果它改善了代码库并遵循项目约定，就批准它。

## 何时使用

- 合并任何 PR 或变更之前
- 完成功能实现后
- 当另一个 agent 或模型产生了你需要评估的代码时
- 重构现有代码时
- 任何 bug 修复后（同时审查修复和回归测试）

## 常见误解

| 误解 | 现实 |
|------|------|
| "它工作了，那就够了" | 不可读、不安全或架构错误的工作代码会产生复合债务。 |
| "我写的，所以我知道它是正确的" | 作者对自己的假设视而不见。每个变更都受益于另一双眼睛。 |
| "我们稍后会清理它" | 后来永远不会到来。审查是质量门禁——使用它。要求在合并前清理，而不是之后。 |
| "AI 生成的代码可能没问题" | AI 代码需要更多审查，而不是更少。它自信且合理，即使在错误时也是如此。 |
| "测试通过了，所以很好" | 测试是必要的但不充分。它们不捕获架构问题、安全性问题或可读性问题。 |

## 红旗

- 没有任何审查就合并的 PR
- 只检查测试是否通过的审查（忽略其他维度）
- 没有实际审查证据的 "LGTM"
- 没有安全重点审查的安全敏感变更
- "太大无法正确审查" 的大 PR（拆分它们）
- 没有回归测试的 bug 修复 PR
- 没有严重性标签的审查评论——不清楚什么是必需的 vs 可选的
- 接受"我稍后会修复它"——它永远不会发生

## 验证

审查完成后：

- [ ] 所有关键问题已解决
- [ ] 所有重要问题已解决或明确推迟并说明理由
- [ ] 测试通过
- [ ] 构建成功
- [ ] 验证故事已记录（更改了什么，如何验证）
