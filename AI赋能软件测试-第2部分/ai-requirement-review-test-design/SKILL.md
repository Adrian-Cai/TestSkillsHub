# AI辅助需求评审与测试点设计

## 技能介绍

这个技能用于把需求文档转成可评审、可追踪、可继续生成用例的测试分析产物。

核心目标：

- 更快看懂需求：提炼业务目标、功能范围和关键流程。
- 更完整拆规则：提取字段、金额、次数、时间、状态、权限等规则。
- 更系统出测试点：覆盖正常流、异常流、边界值、状态变化和兼容/权限风险。
- 更清楚暴露问题：列出需求歧义、隐含假设、待确认项和风险优先级。

## 产出目标（强制）

1. 不能只在对话页面输出结果，必须落地为 Markdown 文件。
2. 默认至少产出 1 份需求评审与测试点设计文档：
   - `outputs/reviews/<slug>.md`
3. 最终回复中必须明确给出产物路径。
4. 如果信息不完整，先继续产出，但必须明确列出所有待确认项和合理假设。

## 输入模式

- 需求文档：PRD、用户故事、产品说明、原型说明、会议纪要。
- 业务流程：登录、支付、订单、优惠券、审批流、退款、积分等功能描述。
- 缺陷修复说明：用于补充回归测试点和影响范围。
- 简短需求：只有自然语言描述时，也要先补齐假设和待确认问题。

## 执行原则

1. 先把需求转成“可验证问题”，不要直接跳到用例。
2. 始终保持“需求输入 -> AI处理 -> 人工审查 -> 测试产出”的闭环。
3. AI 负责初稿、归纳、扩展、检查；测试人员负责业务判断、风险判断和最终确认。
4. 不要把没有依据的内容写成确定规则；无法确认的内容放入“待确认项”。
5. 测试点必须能继续下钻成用例，不能只写“验证功能正常”。

## 标准工作流

复制以下清单并逐步完成：

```text
Task Progress:
- [ ] Step 1: 解析需求输入，生成需求摘要
- [ ] Step 2: 提取业务规则、字段规则、状态规则、权限规则
- [ ] Step 3: 拆解正常流、异常流、边界值和状态变化
- [ ] Step 4: 识别隐含需求、歧义描述和待确认问题
- [ ] Step 5: 生成测试点矩阵并进行风险优先级排序
- [ ] Step 6: 写入 Markdown 产物
```

### Step 1: 生成需求摘要

- 提炼：业务目标、适用用户、功能范围、前置条件、主流程。
- 标出需求来源中的关键原句或关键规则，不要脱离输入材料。

### Step 2: 提取规则清单

至少覆盖：

- 字段规则：必填、格式、长度、枚举、默认值。
- 业务规则：门槛、金额、次数、时间、叠加、互斥、库存。
- 状态规则：创建、使用中、成功、失败、取消、过期、撤回。
- 权限规则：角色、登录态、数据归属、操作权限。

### Step 3: 拆解测试场景

把规则转成可验证场景：

- 正常流：用户按预期路径完成业务。
- 异常流：条件不满足、重复操作、依赖失败、超时、取消。
- 边界值：金额、次数、时间、数量、长度、临界状态。
- 状态变化：状态是否按规则流转，是否允许逆向或重复操作。

### Step 4: 识别隐含需求和待确认项

重点查找：

- 描述模糊：如“满足条件”“有效期内”“优惠后金额”等。
- 规则缺口：未说明叠加、退款、撤销、失败补偿、并发处理。
- 前后矛盾：不同段落对同一规则表达不一致。
- 影响范围：订单、支付、库存、账务、消息通知、数据报表。

### Step 5: 生成测试点矩阵和风险排序

矩阵字段建议：

- 模块/规则
- 测试场景
- 测试类型
- 前置条件
- 测试点
- 预期结果
- 优先级
- 风险说明

风险排序至少考虑：

- 用户影响范围
- 资金/数据/权限风险
- 业务主链路影响
- 历史缺陷或高频变更
- 需求不明确程度

### Step 6: 写入 Markdown 产物

- 目标路径：`outputs/reviews/<slug>.md`
- 文件内容使用“输出模板（默认）”。
- 对话中只做摘要，并给出文件路径。

````markdown
```text

3. 每个测试点给出优先级和风险说明。
````

## 质量门槛

- 必须先输出需求摘要和规则清单，再输出测试点矩阵。
- 测试点不能少于正常流、异常流、边界值、状态变化四类。
- 涉及金额、权限、次数、时间、状态的需求必须单独列规则。
- 每个高风险测试点必须说明风险来源。
- 所有不确定内容必须进入待确认项，不能伪装成需求结论。
- 结果必须落地到文件路径，不能只停留在会话中。
