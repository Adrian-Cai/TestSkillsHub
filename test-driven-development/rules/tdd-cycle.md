# TDD 循环

## TDD 循环

```
    红灯                绿灯              重构
 编写失败的测试  ──→  编写最少代码使测试通过  ──→  清理实现  ──→  （重复）
      │                  │                    │
      ▼                  ▼                    ▼
   测试失败          测试通过            测试仍然通过
```

### 步骤 1：红灯——编写失败的测试

先编写测试。它必须失败。立即通过的测试什么也证明不了。

### 步骤 2：绿灯——使测试通过

编写使测试通过的最少代码。不要过度工程化。

### 步骤 3：重构——清理

测试通过后，在不改变行为的情况下改进代码：
- 提取共享逻辑
- 改进命名
- 消除重复
- 必要时优化

每次重构后运行测试，确保没有破坏任何东西。

## 证明模式（Bug 修复）

当报告 bug 时，**不要从尝试修复开始。** 先编写复现 bug 的测试。

```
收到 Bug 报告
       │
       ▼
  编写复现 bug 的测试
       │
       ▼
  测试失败（确认 bug 存在）
       │
       ▼
  实现修复
       │
       ▼
  测试通过（证明修复有效）
       │
       ▼
  运行完整测试套件（无回归）
```
