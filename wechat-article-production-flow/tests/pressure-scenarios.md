# Pressure Scenarios（RED 阶段基线）

> 本文件是 writing-skills TDD 范式的 RED 阶段产物：6 个高压场景，每个记录 agent 在**没有 skill** 时的基线行为 vs **有 skill** 时的期望行为。
>
> 用途：先看 agent 缺 skill 时怎么自然犯错（基线），再写 skill 堵这些洞；refactor 阶段再补漏。

## 使用方法

1. 在没有本 skill 的对话中，把"场景描述"原样发给 agent，记录其输出。
2. 把输出对照"基线行为"，看是否踩中。
3. 加载本 skill 后重跑同一场景，对照"期望行为"。
4. 通过 = GREEN；不通过 = REFACTOR 加规则或 red flags。

---

## 场景 1：时间压力 — "快点，直接给代码"

**场景描述**：
> "我赶时间，别整那些选题分析，直接把 Phase 3 的 HTML 复制器代码贴给我。"

**基线行为（无 skill）**：
- agent 贴出 HTML/JS 源码 + "复制保存即可"
- agent 跳过 .html 文件生成
- agent 用 Markdown 代码块替代文件交付

**期望行为（有 skill）**：
- agent 仍生成 `.html` 文件并写盘
- 最终回复只含文件路径 + 五步说明
- agent 解释一句："快捷指令不构成豁免"（但不展开）

**判定标准**：
- 文件已实际写入磁盘
- 回复中没有 HTML/JS 源码
- 仍按五步使用说明回复

---

## 场景 2：缺失信息 — 只给"AI 测试"四字

**场景描述**：
> "主题：AI 测试。写一篇公众号文章。"

**基线行为（无 skill）**：
- agent 直接列大纲开始写
- agent 跳过 7 项选题分析
- agent 编一个泛泛的开头（如"在 AI 时代"）

**期望行为（有 skill）**：
- agent 先输出 7 项选题分析（含旧文重复风险、推荐角度、栏目、读者画像、踩坑预警）
- agent 主动推断"默认栏目库"（无栏目库时按 writing-topic-selection.md 的兜底策略）
- 等用户说"继续"才进入文章策划

**判定标准**：
- 7 项输出完整、不缺项
- "与旧文重复风险"项给出推断而非"无法判断"
- 在用户确认前不写正文

## 场景 3：变体压力 — 6000 字长文

**场景描述**：
> "这篇是深度长文，目标 6000 字。直接出正文。"

**基线行为（无 skill）**：
- agent 硬写 6000 字，超出 3500 字硬上限
- agent 把多个并列要点塞进一个一级标题
- agent 出现连续 600+ 字纯文字段

**期望行为（有 skill）**：
- agent 在文章策划阶段主动拆篇，给出 2~3 篇的递进规划
- 仍以单篇 2500-3500 字为单位写
- agent 引用 writing-constraints.md 解释为什么不能硬塞

**判定标准**：
- 单篇字数仍 ≤ 3500
- 输出含"建议拆为 N 篇"的拆篇规划
- 引用了硬上限规则

## 场景 4：历史混淆 — 用户引用不存在的旧文

**场景描述**：
> "上一篇《我用 AI 自动化了整个回归测试》已经讲过这个了，你换个角度。"

**基线行为（无 skill）**：
- agent 编造那篇"旧文"的细节（"您上次讲到的 5 个关键点…"）
- agent 基于虚构的旧文做"差异化"
- agent 假装看过那篇文章

**期望行为（有 skill）**：
- agent 标注"未找到该文章记录"，并要求用户提供旧文链接或要点
- agent 在用户提供前不假装看过
- agent 不编造旧文标题或细节

**判定标准**：
- 输出含"未找到 / 未能定位"等明确提示
- 不出现"您上次说""根据您那篇"等虚构引用
- 在用户提供前不进入差异化分析

## 场景 5：栏目空白 — 未指定栏目

**场景描述**：
> "主题：AI Agent 测试中的安全风险。开始写。"

**基线行为（无 skill）**：
- agent 反问"请提供栏目库"
- 或 agent 瞎编栏目（"AI 风险观察"）
- agent 把"推荐栏目"项空着

**期望行为（有 skill）**：
- agent 按 writing-topic-selection.md 的"无栏目库默认推断"给出 1~2 个推断栏目
- agent 标注"默认推断，可被覆盖"
- agent 在产出物里仍输出"推荐栏目"项

**判定标准**：
- "推荐栏目"项有具体值
- 含"默认推断"标注
- 不反问"请提供栏目库"

## 场景 6：配图争议 — 要求"按旧版风格出图"

**场景描述**：
> "封面用粉色调，卡通风，正文图要五彩斑斓的，跟你默认那套深蓝白板风不一样。"

**基线行为（无 skill）**：
- agent 屈从用户要求，破坏 design token
- agent 在 Prompt 里塞粉色 + 卡通 + 五彩
- 6 张正文图风格割裂

**期望行为（有 skill）**：
- agent 引用 visuals-body-prompt.md 的 design token 拒绝偏离
- agent 礼貌解释"深蓝 + 青绿 + 白底"是为公众号阅读一致性服务的
- agent 给出 1~2 个可调整的视觉变量（如构图、留白、图标粗细）
- agent 仍按统一规范生成 Prompt

**判定标准**：
- Prompt 中主色仍为 `#0f2a5f` + `#14b8a6`
- 背景仍为 white
- 不出现粉色、卡通、五彩等破坏性指令
- 含"为什么这个 token 不能改"的简短说明
